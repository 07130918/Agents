# principle-of-programming-reviewer

この参照は複数 CLI から使う共通本体です。実行中の CLI に対応する節だけを使用してください。

## Codex / .agents 版

# /principle-of-programming-reviewer (/popr)

現在の作業ブランチの変更を「Principle of Programming」の原則に基づいてレビューする。言語・技術に依存しない普遍的な観点で評価し、具体的な改善提案を出す。エイリアス: `/popr`。

## 実行方法

```bash
# フルコマンド
/principle-of-programming-reviewer

# 短縮コマンド
/popr
```

## 手順

1. ベースブランチを判定 (`develop` 優先、無ければ `main`)
2. `git diff <base>...HEAD --no-color` で差分を取得し、変更ファイルを Read
3. 下記 7 カテゴリで評価し、具体的なコード箇所を `file:line` 形式で引用しながら分析
4. 出力フォーマットに従って結果を提示

## レビュー観点 (7 カテゴリ)

軸名のみ列挙する。
各原則の中身は Codex が既知。**コードの該当箇所と原則名を結びつけて指摘する** のがこの skill の役割。

### 1. 前提の確認
- コード自体がドキュメントになっているか
- 複雑性を受け入れつつ、必要以上に複雑にしていないか
- 自己文書化コードの実現度

### 2. 原則の遵守
- **KISS原則**: 修正容易性、シンプルな設計
- **DRY原則**: 重複の排除、知識の一元化
- **単純性 > 汎用性**: 過度な抽象化の回避
- **可読性重視**: 理解しやすいコード
- **抽象度の統一**: 関数内の一貫性
- **命名の明確性**: 効果と目的を説明する名前

### 3. 思想の反映
- **セオリー**: コミュニケーション、シンプル、柔軟性
- **ロジックとデータの近接性**: 関連する処理とデータの配置
- **カプセル化・情報隠蔽**: 実装詳細の隠蔽
- **関心の分離**: 責務の明確な分離
- **ポリシーと実装の分離**: ビジネスルールと実装の分離
- **UNIX思想17原則**: モジュール性、明確性、簡潔性、透明性など
- **UNIX哲学9項目**: 小さくシンプルなプログラム、連携など

### 4. 視点の評価
- **凝集度**: モジュール内の関連性の強さ
- **結合度**: モジュール間の独立性
- **冪等性と安全性**: 副作用の管理
- **直交性**: 独立性と分離性
- **可逆性**: 特定技術への依存度
- **コードの臭い**: アンチパターンの検出
- **技術的負債**: 将来のコスト評価

### 5. 習慣の実践
- **ボーイスカウトの規則**: 来た時より綺麗にする
- **パフォーマンス < 可読性**: 最適化より理解しやすさ
- **1度に1つの作業**: 単一責任の徹底

### 6. 手法の適用
- **契約による設計**: 前提条件・事後条件の明確化
- **防御的プログラミング**: ガード節、エラーハンドリング
- **コンテキストの明確化**: 状態と依存関係の可視化

### 7. 法則の考慮
- **割れた窓の法則**: 悪いコードの放置防止
- **エントロピーの法則**: コードの腐敗への対処
- **セカンドシステム症候群**: 過剰な機能追加の回避
- **車輪の再開発**: 既存ソリューションの活用

## 出力フォーマット

```markdown
# Principle of Programming レビュー結果

## 総合評価
[A-F の評価 (A=優秀, B=良好, C=普通, D=要改善, F=大幅改善)]
[全体的な品質・保守性・拡張性の総括]

## ✅ 優れている点
- `path/to/file.ts:line_number` — [守られている原則名] — [なぜ優れているか]

## ⚠️ 改善が必要な点
- `path/to/file.ts:line_number` — [違反している原則名] — [改善コード例]

## 優先的な改善アクション
1. [最優先] [品質・保守性への影響大]
2. [高優先] [次回リファクタリングで対応]
3. [中優先] [長期的に改善]
4. [低優先] [将来的に検討]
```

## 指針

- ✅ 言語・技術に依存しない普遍原則で評価する
- ✅ 批判だけでなく優れている点も必ず指摘する
- ✅ 改善提案は具体的かつ実行可能な形で提示する
- ✅ コード参照は `file_path:line_number` 形式で行う
- ✅ 建設的なトーンで記述(非難ではなく改善支援)
- 変更ファイル数が多い場合は主要ファイルに焦点を当てる
