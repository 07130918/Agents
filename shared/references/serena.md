# /serena

Serena方式でアプリ開発、デバッグ、設計、レビューを構造化して進める。Serena MCP tools が使える場合は優先して使い、未接続なら通常の検索、読解、編集ツールで同じワークフローを実行する。

## 使う場面

- `/serena`、`serena`、`serena-dev` が明示された依頼。
- 実装、バグ調査、設計、リファクタ、レビュー、最適化を手順化して進めたい依頼。
- 複数ファイルや既存設計への影響があり、先に調査と方針決定が必要な依頼。

対象外:

- 単純な質問、短い翻訳、1コマンドで済む確認。
- PR作成、CI調査、セキュリティ監査など、専用 skill が明らかに優先される依頼。

## クイックリファレンス

```bash
/serena <問題> [オプション]
/serena debug "テストがpassしない"
/serena design "新規エンドポイントの設計"
/serena implement "検索フォームを改善"
/serena review "この差分をレビュー"
```

## オプション

| オプション | 効果 | 使用ケース |
|---|---|---|
| `-q` | 調査範囲を絞って素早く進める | 軽微な修正、明確な原因のあるバグ |
| `-d` | 調査深度を上げ、代替案とリスクも確認する | 複雑な障害、重要な設計判断 |
| `-c` | コードと差分を中心に確認する | 実装、リファクタ、レビュー |
| `-s` | 手順を分けて進め、必要ならチェックリストを使う | 複数ステップの機能開発 |
| `-v` | 判断根拠、確認結果、次の作業を明示する | 学習、レビューしやすい進行 |
| `-r` | 公式ドキュメントや現行仕様の確認を含める | 技術選定、最新情報が必要な作業 |
| `-t` | 実装TODOを作る | 中規模以上の作業管理 |

互換オプション:

- `--max-thoughts=N`: 旧形式。新規では使わず、調査深度の目安としてだけ扱う。
- `--focus=AREA`: frontend、backend、database、security など重点領域を指定する。
- `--context=FILES`: 指定ファイルを先に読む。
- `--summary`: 最終出力を要約中心にする。
- `--progressive`: 進捗更新を細かくする。

## タスク分類

ユーザー指定があればそれを優先する。指定がなければキーワードと文脈から分類する。

| 分類 | キーワード例 | 主要成果 |
|---|---|---|
| debug | error, bug, broken, failing, エラー, バグ, 落ちる, 失敗, 動かない, 壊れている, 直して | 根本原因、修正、回帰確認 |
| design | architecture, design, system, plan, 仕様, 設計, 方針, 分割, アーキテクチャ | 選択肢、判断、実装計画 |
| implement | build, create, add, feature, implement, 実装, 追加, 作成, 改善, 対応 | 実装、検証、変更要約 |
| review | review, analyze, performance, refactor, slow, レビュー, 評価, リファクタ, 遅い, 最適化 | 指摘、リスク、改善案 |

曖昧な場合は、既存コードを少し読んで分類を決める。分類だけで止まらず、依頼が実装なら最後まで編集と検証まで進める。

## 共通手順

1. 依頼を debug / design / implement / review に分類する。
2. 既存コード、設定、テスト、ドキュメントから関連する規約を確認する。
3. Serena MCP tools が使える場合は、シンボル概要、参照検索、パターン検索を優先する。
4. Serena MCP tools が使えない場合は、`rg`、`rg --files`、通常のファイル読解、通常の編集ツールで代替する。
5. 変更前に、原因、設計判断、または実装方針を短く明示する。
6. 既存パターンに沿って最小限の変更を行う。
7. 変更範囲に見合うテスト、型チェック、lint、手動確認を実行する。
8. 最終報告で実施内容、変更ファイル、検証結果、残リスクを簡潔にまとめる。

## 分類別チェックリスト

### debug

1. 症状、再現手順、失敗ログを確認する。
2. 環境変数、設定、キャッシュ、依存関係を早めに確認する。
3. 仮説を複数出し、証拠で絞る。
4. 根本原因を直す。症状だけを抑える修正で終わらせない。
5. 回帰テストまたは再現確認を追加する。

### design

1. 要件、制約、成功条件を整理する。
2. 既存アーキテクチャと責務境界を確認する。
3. 2から3案を比較し、採用案と不採用理由を明示する。
4. 実装順序、検証方法、移行リスクを決める。
5. 必要なら設計ドキュメントやADRに残す。

### implement

1. 既存の命名、配置、データフロー、テスト規約を読む。
2. 変更範囲を絞り、不要な抽象化を追加しない。
3. 実装、テスト、検証を同じ流れで進める。
4. 共有処理を作る場合は、実際の重複や複雑性を減らす時だけにする。
5. ユーザーや他作業の無関係な変更を巻き戻さない。

### review

1. 差分と周辺コードを確認する。
2. 正確性、セキュリティ、保守性、テスト不足を優先して見る。
3. 指摘はファイルと行に紐づけ、重大度順に並べる。
4. 好みの問題だけなら明確に区別する。
5. 修正依頼がある場合だけ編集する。

## 出力

- 途中更新は、今読んでいる文脈と次の作業を短く伝える。
- `-v` 指定時も内部推論の全文は出さず、判断根拠、確認結果、次の作業を出す。
- 最終報告は、変更内容、検証結果、未確認事項を中心にする。
- 実装していない提案は、実装済みと混ぜない。
