# /serena

Serena MCP を使った構造化問題解決。問題タイプを自動検出し、適切な思考回数で進める。

## クイックリファレンス

```bash
/serena <問題> [オプション]
/serena debug "テストがpassしない"          # デバッグ (5-8 思考)
/serena design "新規エンドポイントの作成"     # 設計 (8-12 思考)
/serena review "このコードを最適化して"       # レビュー (4-7 思考)
/serena implement "機能Xを追加"              # 実装 (6-10 思考)
```

## オプション

| オプション | 説明 | 使用ケース |
|---|---|---|
| `-q` | クイック (3-5 思考) | 簡単なバグ、マイナー機能 |
| `-d` | 深思考 (10-15 思考) | 複雑なシステム、重要な決定 |
| `-c` | コード重視 | コードレビュー、リファクタリング |
| `-s` | ステップバイステップ実装 | フル機能開発 |
| `-v` | 詳細出力 (プロセス表示) | 学習、プロセス理解 |
| `-r` | リサーチフェーズを含む | 技術選定 |
| `-t` | 実装 TODO を作成 | プロジェクト管理 |

複合例: `/serena "マイクロサービス設計" -d -r -v` / `/serena "ダッシュボード実装" -s -t -c`

## 高度なオプション

- `--max-thoughts=N`: 思考回数を上書き
- `--focus=AREA`: ドメイン特化 (frontend/backend/database/security)
- `--token-budget=N`: トークン制限
- `--context=FILES`: 特定ファイルを先に分析
- `--summary` / `--progressive`: 出力モード

## 問題検出と思考回数

キーワードから自動選択する:

- **デバッグ** (error, bug, broken, failing): 5-8 思考
- **設計** (architecture, system, structure, plan): 8-12 思考
- **実装** (build, create, add, feature): 6-10 思考
- **最適化/レビュー** (analyze, performance, refactor, slow): 4-7 思考

## 問題別テンプレート

### デバッグ (5-8 思考)
1. 症状分析と再現
2. エラーコンテキスト・環境チェック
3. 根本原因の仮説生成
4. 証拠収集と検証
5. 解決策設計とリスク評価
6. 実装計画
7. 検証戦略
8. 予防措置

### 設計 (8-12 思考)
1. 要件の明確化
2. 制約と前提条件
3. ステークホルダー分析
4. アーキテクチャオプション生成
5. オプション評価 (長所/短所)
6. 技術選定
7. 設計決定とトレードオフ
8. 実装フェーズ
9. リスク軽減
10. 成功指標
11. 検証計画
12. ドキュメントニーズ

### 実装 (6-10 思考)
1. 機能仕様とスコープ
2. 技術的アプローチ選択
3. コンポーネント/モジュール設計
4. 依存関係と統合ポイント
5. 実装順序
6. テスト戦略
7. エッジケース処理
8. パフォーマンス考慮
9. エラー処理とリカバリー
10. デプロイ・ロールバック計画

### レビュー/最適化 (4-7 思考)
1. 現状分析
2. ボトルネック特定
3. 改善機会
4. 解決策と実現可能性
5. 実装優先度
6. パフォーマンス影響推定
7. 検証・監視計画

## 実行ガイドライン

1. 問題タイプを自動検出してテンプレート選択
2. Serena MCP のセマンティックコード検索・編集を活用
3. 問題分析から始めて具体的アクションで終わる
4. `-s` 指定時は実装 TODO を作成
5. セキュリティ・パフォーマンス・保守性を考慮

## トークン効率のヒント

- 簡単な問題には `-q` (約 40% 削減)
- 概要のみが必要なら `--summary`
- 関連する問題は単一セッションに統合
- 無関係な分析を避けるなら `--focus`
