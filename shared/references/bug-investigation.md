# バグ調査ワークフロー

## 基本方針

「症状を修正する」のではなく「根本原因を特定する」こと。修正案を出す前に、必ず調査フェーズを完了させる。

ハードなバグでは、ステップを飛ばして良いのは**明示的に正当化できる時だけ**。とくに Step 0 (フィードバックループ構築) は飛ばさない。

---

## 調査の手順

### Step 0: フィードバックループの構築 (最重要)

**これが本スキルの核心**。残りのすべてのフェーズは機械作業で、ループさえあれば bisection・仮説検証・計装は信号を消費するだけで原因にたどり着ける。ループが無いと、どれだけコードを睨んでも原因は見えない。

**ここに不釣合いなほど時間を投じる。粘り強く、創造的に、諦めない。**

#### ループの作り方 (おおむねこの順で試す)

1. **失敗するテスト** をバグに到達する seam (unit / integration / e2e) で書く
2. **curl / HTTP スクリプト** をローカル開発サーバーに叩く
3. **CLI 起動 + フィクスチャ入力** + stdout を既知の正解 snapshot と diff
4. **ヘッドレスブラウザ** (Playwright / Puppeteer) で UI を駆動し、DOM/console/network を assert
5. **キャプチャしたトレースを再生** — 実際のリクエスト/ペイロード/イベントログをファイルに保存し、コードパスに単独で流し込む
6. **使い捨てハーネス** — システムの最小サブセット (1 サービス + モック) を立ち上げ、バグコードパスを 1 関数呼び出しで再現
7. **Property / fuzz ループ** — 「たまに出力が間違う」系なら 1000 回ランダム入力を流し、失敗モードを探す
8. **Bisection ハーネス** — 既知の良/悪状態 (commit / dataset / version) の間で発生したなら、`git bisect run` できる pass/fail スクリプトを書く
9. **Differential ループ** — 同じ入力を旧版 vs 新版 (または 2 つの設定) に流して diff を取る
10. **HITL bash スクリプト** — 最終手段。人間がクリックする必要がある場合は、任意の場所にスクリプトを作る (例: `scripts/hitl-loop.sh`) 形で手順を構造化し、入力を捕捉してエージェントに戻す

**正しいループが組めれば、バグは 90% 直ったも同然**。

#### ループ自体を磨く

ループは製品として扱う。1 本でき次第、自問する:

- 速くできるか? (セットアップをキャッシュ、無関係な init をスキップ、テストスコープを絞る)
- シグナルを鋭くできるか? (「クラッシュしない」ではなく「特定の症状」を assert)
- 決定論的にできるか? (時刻固定、RNG seed、ファイルシステム隔離、ネットワーク凍結)

**30 秒の flaky なループは、ループが無いのとほぼ同じ**。2 秒の決定論的ループはデバッグの超能力。

#### 非決定論バグ

目標は「綺麗な再現」ではなく「**再現率を上げる**」こと。トリガを 100 回回す、並列化、ストレス、タイミングウィンドウを狭める、sleep を注入する。**1% のフレークは debuggable ではない、50% まで上げてから先に進む**。

#### ループが本当に組めない時

明示的に止まり、何を試したかをユーザに伝え、次のいずれかを依頼する:

- 再現する環境へのアクセス
- キャプチャされた成果物 (HAR / ログダンプ / コアダンプ / タイムスタンプ付きスクリーン録画)
- 一時的に本番計装を入れる許可

ループ無しで Step 4 (仮説) に進んではいけない。

### Step 1: 症状の明確化

- エラーメッセージの**完全なテキスト**を取得する
- 再現条件を特定する (常時? 特定のユーザー? 特定の操作?)
- 最後に動作した時点を特定する (いつから壊れた?)
- 関連するファイル・モジュールを初期把握する
- **Step 0 のループが「ユーザが言っているバグ」を再現できているか**を必ず確認する。近くで起きている別の失敗を直しても、間違ったバグを直したことになる

### Step 2: ログ収集

- ブラウザコンソールのエラーを確認する
- サーバログ (Flask / FastAPI / Next.js) を確認する
- DB クエリログを確認する
- ネットワークタブでの API レスポンスを確認する

### Step 3: コード調査

- エラーメッセージからファイル・行番号を特定する
- `git log --oneline -20` で最近の変更を確認する
- `git diff HEAD~5` で変更内容を確認する
- 環境変数・設定ファイルを確認する
- 該当領域のドメイン用語集 (`CONTEXT.md`) や `docs/adr/` があれば先に読む

### Step 4: 仮説のランク付け (3-5 個)

**最初に思いついた 1 つに固執しない**。3-5 個の候補を順位付きで生成してから検証に進む。各仮説には **falsifier (反証可能宣言)** を必須化する。

> フォーマット: 「もし **X** が原因なら、**Y を変えればバグが消える** / **Z を変えれば悪化する**」

予測が書けないなら、その仮説は単なる勘 — 捨てるか研ぎ直す。

#### よくある原因パターン (発生頻度順、karte-web / uka-route 文脈)

1. **環境変数の未設定/誤設定** — `.env` ファイルの確認
2. **Chakra UI v3 prop 変更** — `isDisabled` → `disabled` 等のマイグレーション漏れ
3. **SQL バグ** — カンマ漏れ、JOIN 条件ミス、NULL 扱いのミス
4. **型の不一致** — TypeScript 型エラー、API レスポンス形式の変化
5. **キャッシュ不整合** — Next.js `.next` / ブラウザ / Redis キャッシュ
6. **権限・ロール判定の漏れ** — 新しいロールに対応していない条件分岐
7. **非同期処理の問題** — `await` 漏れ、Promise のエラーハンドリング不足

#### ユーザに見せる

ランク付き仮説リストはテスト前にユーザに提示する。ドメイン知識で即座に並び替えられる (「3 番目はちょうど昨日デプロイした変更だ」「2 番目は先週検証済み」など) ことが多い。**安いチェックポイントで大きく時間が節約できる**。ユーザが AFK ならそのまま自分のランクで進める。

### Step 5: 計装と検証

各プローブは Step 4 のいずれかの予測に対応させる。**変数は一度に 1 つだけ変える**。

#### ツール選択

1. **デバッガ / REPL inspection** が使える環境なら最優先。1 つのブレークポイントは 10 個のログに勝る
2. **境界に絞ったログ** で仮説を区別する
3. **「全部ログ出して grep」は禁止**

#### DEBUG タグ規律

すべてのデバッグログに `[DEBUG-a4f2]` のような**ユニーク prefix** を必ず付ける。後でクリーンアップする時、grep 1 発で全削除できる。タグなしログは生き残ってしまう。

#### パフォーマンス回帰の場合

ログは大抵間違っている。代わりに:

- ベースライン測定を確立 (タイミングハーネス、`performance.now()`、プロファイラ、クエリプラン)
- bisect する
- **計測してから直す。逆ではない**

### Step 6: 修正と回帰テスト

回帰テストは**修正の前に**書く。ただし**正しい seam がある場合に限る**。

正しい seam とは、テストが**実際のバグパターンを呼び出し元と同じ形で**動かす場所。シャローすぎる seam (バグは複数 caller の連鎖で起きるのに、シングル caller でしかテストできない) で書いた回帰テストは偽の安心感を与えるだけ。

**正しい seam が無い場合、それ自体が finding**。記録し、Step 7 のポストモーテムに渡す。コードベースのアーキテクチャがバグを固定できなくしているサインで、`software-architecture` skill にハンドオフ候補。

正しい seam がある場合:

1. 最小化された再現を、その seam での失敗するテストに変換
2. テストが失敗するのを確認
3. 修正を当てる
4. テストが通るのを確認
5. **Step 0 のフィードバックループを未最小化のシナリオで再実行** して全体での修正を確認

### Step 7: クリーンアップとポストモーテム

完了宣言の前に必須:

- [ ] 元の再現が再現しないこと (Step 0 のループを再実行)
- [ ] 回帰テストが通ること (または seam の不在が記録されていること)
- [ ] `[DEBUG-...]` 計装をすべて削除 (`grep` で確認)
- [ ] 使い捨てプロトタイプを削除 (またはデバッグ用と明記された場所に移動)
- [ ] **正解だった仮説を commit / PR メッセージに残す** — 次のデバッガが学べるように

最後に「**これを防ぐにはどんなアーキテクチャ変更が必要か?**」を問う。回答が「テスト seam が無い」「caller が絡まっている」「隠れた結合がある」なら `software-architecture` skill に具体を添えてハンドオフする。**修正後に推薦すること** (修正前より情報量が多くなっているので判断精度が上がる)。

---

## 修正前チェックリスト

- [ ] 根本原因が特定できているか (症状ではなく原因)
- [ ] 同様の箇所が他にないか (Grep で確認)
- [ ] 修正がサイドエフェクトを持たないか
- [ ] テストケースが存在するか/追加すべきか
- [ ] マイグレーションが必要な DB 変更はないか

## 再現手順の記録フォーマット

```
## バグ報告: [タイトル]

**環境**: 開発/ステージング/本番
**発生条件**: [具体的な操作手順]
**期待する動作**: [何が起きるべきか]
**実際の動作**: [何が起きているか]
**エラーログ**: [完全なエラーメッセージ]
**関連ファイル**: [調査対象のファイルパス]
**フィードバックループ**: [Step 0 で組んだ pass/fail スクリプト or テストコマンド]
```

- DB マイグレーションがらみ: `python-backend` / `nextjs-prisma-patterns`
