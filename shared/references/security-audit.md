# security-audit

この参照は複数 CLI から使う共通本体です。実行中の CLI に対応する節だけを使用してください。

## Codex / .agents 版

# セキュリティ監査ワークフロー

技術スタック非依存。10 ラウンドの反復で網羅性を確保し、重み付きスコアで定量評価する。

## 詳細リファレンス

| ファイル | 内容 | 読むタイミング |
|---|---|---|
| `reference-attack-surface.md` | 6 カテゴリの攻撃面チェックリスト (認証・認可・CSRF・入力検証・インフラ・ログ) | Phase 2 探索時、Phase 5 新規発見時 |
| `reference-scoring.md` | 6 カテゴリ重み付きスコア、深刻度別減点、レポート構造テンプレート | Phase 6 スコア計算時、最終レポート作成時 |

## 基本方針

- 「レポートを信じる」のではなく「コードで検証する」
- 全ての指摘事項はファイルパス・行番号・コードスニペットで裏付ける
- 10 ラウンド以上の反復検証で網羅性を確保
- 技術スタックに依存しない汎用観点でチェック

## 10 フェーズのワークフロー

### Phase 1: 既存レポートの読み込み or 初期スキャン

**既存レポートあり**: 全体を読み、各指摘の (ID, 深刻度, カテゴリ, ファイルパス, 行番号, コードスニペット) を記録。

**新規監査**: プロジェクト構成を把握 (ディレクトリ、技術スタック、依存関係)。AGENTS.md / README.md からアーキテクチャ取得。`package.json` / `requirements.txt` / `go.mod` で依存確認。

### Phase 2: コードベース探索 (攻撃面の特定)

`reference-attack-surface.md` の 6 カテゴリを順に探索:
1. 認証・セッション管理
2. 認可・アクセス制御
3. CSRF・トランスポートセキュリティ
4. 入力検証・インジェクション
5. インフラ・サーバー設定
6. ログ・監視・情報漏洩

### Phase 3: 指摘事項の検証 (既存レポートある場合)

各指摘について:
1. **ファイルパス存在確認** (Glob)
2. **行番号の正確性** (Read)
3. **コードスニペットの一致** (レポート vs 実コード)
4. **攻撃シナリオの妥当性** (実際に成立するか)
5. **深刻度の適切性** (CVSS/OWASP に照らす)

#### よくある誤りパターン

- ❌ リダイレクト先がハードコードなのに「オープンリダイレクト」と記述
- ❌ 1 テーブルだけ言及だが実際は複数テーブル影響
- ❌ ORM が自動エスケープしているのに「SQLi」と記述
- ❌ CSP 設定済みなのに「XSS 脆弱性」と断定

### Phase 4: 修正・降格・昇格

- **修正**: 攻撃シナリオの書き換え、影響範囲の訂正
- **降格**: 実際の影響が軽微な場合 (Critical → High)
- **昇格**: 実際の影響がレポートより深刻
- **範囲拡大**: 同じ脆弱性が他箇所にも存在する場合

### Phase 5: 新規脆弱性の発見

レポート未記載のものを `reference-attack-surface.md` のチェックリストで探索。代表例:

- DB マイグレーション履歴の整合性 (UNIQUE 制約の回帰、PK 喪失)
- ボディパーサーのサイズ制限 (DoS)
- localStorage / sessionStorage の機密データ
- trust proxy 未設定による IP 偽装
- デバッグ用プラグイン/ミドルウェアの本番残存
- ORM スキーマと DB の乖離
- 依存パッケージのパッチファイルによるセキュリティ変更
- 開発サーバーのネットワーク公開設定 (`host: true`)
- トランザクション未使用の複数 DB 書き込み (TOCTOU)
- 環境変数の空文字列デフォルト

### Phase 6-10: 反復更新

- **Round 6**: スコア再計算 (`reference-scoring.md`)
- **Round 7**: ファイル別脆弱性インデックス更新
- **Round 8**: 修正ロードマップ更新
- **Round 9**: エグゼクティブサマリー更新
- **Round 10**: 最終整合性チェック

## 検証の品質基準

修正・追加した全指摘が以下を満たすこと:

- [ ] ファイルパスが実在する (Glob)
- [ ] 行番号が正確 (Read)
- [ ] コードスニペットが実コードと一致
- [ ] 攻撃シナリオが論理的に成立
- [ ] 深刻度が影響に見合う
- [ ] 推奨対策が実装可能
- [ ] 脆弱性 ID がユニーク
- [ ] スコア計算が数学的に整合
- [ ] ファイル別インデックスが全脆弱性を網羅
- [ ] ロードマップの Phase が深刻度と一致

## 使用方法

### 既存レポートの検証

```
このプロジェクトの SECURITY_AUDIT_REPORT.md を検証してください。
各指摘をコードと照合し、誤りの修正・新規脆弱性の追加・スコア再計算を行ってください。
```

### 新規監査の実施

```
このプロジェクトのセキュリティ監査を実施してください。
6 カテゴリの重み付きスコアリングで定量評価し、レポートを作成してください。
```
