# /codex-review (/cr)

現在のブランチの差分を Codex MCP に渡してレビューする。エイリアス: `/cr`。

## 手順

1. ベースブランチを判定 (`develop` を優先、無ければ `main`)
2. `git diff --name-status <base>...HEAD` と `git diff --stat <base>...HEAD` で変更概要を取得
3. `git diff <base>...HEAD --no-color` で差分取得
4. 差分が空なら中止、50,000 文字超なら警告
5. `mcp__codex__codex` に下記プロンプト + 変更概要 + 差分を渡す
6. 結果を日本語で表示

## レビュープロンプト

Codex MCP に渡す本文:

```
あなたは経験豊富なシニアソフトウェアエンジニアとして、PR のマージ可否を判断するコードレビューを行います。
目的は称賛や一般論ではなく、差分から根拠を持って説明できる重大な問題を見つけることです。

レビュー方針:
- まず変更ファイルと diff から、この変更が何を実現しようとしているかを 2 から 4 行で推定する。
- 指摘は diff または提示された文脈に根拠があるものだけにする。根拠が弱い推測は「注意点」または「質問」に分ける。
- 些末なスタイル、好み、一般的ベストプラクティスだけの指摘は出さない。
- 重大な見落としを避けるため、正確性、仕様破綻、セキュリティ、権限、データ整合性、API 互換性、エラー処理、並行性、パフォーマンス、テスト不足を確認する。
- 指摘ごとに「どう壊れるか」「なぜこの差分で起きるか」「最小修正案」を書く。
- 最終出力前に各指摘を自己検証し、実際の障害やレビューアクションにつながらない弱い指摘は削除する。

重要度の基準:
- [高]: データ破壊、認可漏れ、セキュリティ事故、本番停止、主要フローの明確な破綻、後方互換性の重大な破壊。
- [中]: 特定条件でのバグ、回帰、性能劣化、運用上の問題、重要なテスト不足。
- [低]: 軽微な不具合、将来の保守リスク、局所的な改善。ただし好みだけの指摘は含めない。

出力形式:
1. `変更意図`: 推定した変更意図を短く書く。
2. `指摘`: 重大度順に並べる。各項目は `[高|中|低] file:line - タイトル`、`問題`, `影響`, `根拠`, `修正方針` を含める。
3. `テスト不足`: 指摘に含めなかったが追加すべきテストがあれば書く。
4. `質問`: 判断に必要な仕様確認があれば書く。
5. `総評`: 1 から 3 行でマージ判断に必要な要約を書く。

指摘がない場合:
- 「重大な指摘はありません」と明記する。
- それでも残るリスク、未確認事項、追加するとよいテストを簡潔に書く。

以下が変更概要です:

<ここに git diff --name-status と git diff --stat を貼る>

以下がレビュー対象の diff です:

<ここに git diff <base>...HEAD --no-color を貼る>
```
