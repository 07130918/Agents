# ADR フォーマット

`grill-with-docs` の grilling セッション中に、`docs/adr/` に Architecture Decision Record を提案する時の形式と判断基準。

参考: Matt Pocock "grill-with-docs/ADR-FORMAT.md" を日本語化。

---

## 配置と採番

ADR は `docs/adr/` に置く。連番命名: `0001-slug.md`, `0002-slug.md`, …

`docs/adr/` ディレクトリは**遅延作成**する — 最初の ADR が必要になった時に作る。

採番は `docs/adr/` をスキャンし、最大の既存番号 + 1 を使う。

---

## テンプレート

```md
# {決定の短いタイトル}

{1-3 文: コンテキスト・何を決めたか・なぜ}
```

これだけ。**ADR は 1 段落で十分**。価値は「決定が行われたこと」と「その理由」を残すことにある。section を埋めることに価値があるわけではない。

---

## 任意セクション

以下は**真に価値を加える時だけ**含める。大半の ADR には不要。

- **Status** frontmatter (`proposed | accepted | deprecated | superseded by ADR-NNNN`) — 決定が再検討される時に有用
- **Considered Options** — 却下した代替案を**覚えておく価値がある**時のみ
- **Consequences** — 自明でない下流効果をコールアウトする必要がある時のみ

---

## ADR をいつ提案するか

**3 条件すべて**を満たす時だけ:

1. **元に戻しにくい** — 後で考えを変えるコストが意味のあるレベル
2. **コンテキスト無しでは驚く** — 将来の読み手が「なぜこの実装にしたのか?」と思う
3. **真のトレードオフの結果** — 真の代替があり、特定の理由でそちらを選んだ

決定が**簡単に元に戻せる**ならスキップ — どうせ戻す時に戻す。**驚くものでない**なら誰も理由を疑問に思わない。**真の代替が無かった**なら「自明な選択をした」以上に記録するものが無い。

---

## 該当する典型例

- **アーキテクチャの形**: 「monorepo を採用」「書き込みモデルは event-sourced、読み込みモデルは Postgres に projection」
- **コンテキスト間の統合パターン**: 「Ordering と Billing は domain event で通信、同期 HTTP は使わない」
- **lock-in を伴う技術選択**: DB、メッセージバス、認証プロバイダ、デプロイターゲット。**全てのライブラリではなく**、「四半期かけないと差し替えできない」ものだけ
- **境界とスコープの決定**: 「Customer データは Customer コンテキストが所有、他コンテキストは ID 経由で参照のみ」。**明示的な「やらない」**は「やる」と同じくらい価値がある
- **意図的に当然路から外れた決定**: 「ORM ではなく手動 SQL を使う、理由は X」。合理的な読み手が逆を仮定するもの。次のエンジニアが「修正」しないように残す
- **コードに見えない制約**: 「コンプライアンス上 AWS は使えない」「パートナー API 契約上、応答は 200ms 未満」
- **却下した代替案 (却下理由が自明でない時)**: GraphQL を検討して REP を選んだ理由が subtle なら記録する。しないと半年後に誰かが GraphQL を再提案する

---

## 該当しない例 (=ADR を作らない)

- 「ライブラリ X を使った」(差し替えコスト低)
- 「コードフォーマッタは Prettier」(自明な選択肢)
- 「コミットメッセージは英語」(規約に近い)
- 「変数命名は camelCase」(コードスタイル、ADR ではなく lint ルールへ)

迷ったら**書かない**。冗長な ADR は本物の ADR を埋もれさせる。
